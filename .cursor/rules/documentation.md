# Стандарты документации

## Когда документировать (JSDoc и аналоги)

**Документируй:**

- Публичные API
- Сложные алгоритмы
- Неочевидную бизнес-логику
- Обходные решения и хаки (с TODO на исправление)

**Можно не документировать:**

- Самодокументируемый код
- Приватные вспомогательные функции
- Простые CRUD-операции
- Геттеры/сеттеры

## Формат

```typescript
/**
 * Аутентифицирует пользователя по email и паролю.
 * 
 * @param email - Email пользователя
 * @param password - Пароль в открытом виде
 * @returns Объект пользователя с токеном сессии
 * @throws {AuthError} Если учётные данные неверны
 */
async function login(email: string, password: string): Promise<AuthResult> {
  // ...
}
```

## Когда использовать skill docs

**Читай `.cursor/skills/docs/SKILL.md`, когда:**

- пользователь просит обновить или создать документацию
- после реализации фичи
- нужно понять структуру документации проекта (пути в `.cursor/config.json`)
- пользователь говорит про документацию, docs или просит «задокументировать это»
