## Цель и критерии успеха

- В `main` нет `ai_docs`, `.venv`, `.pytest_cache`, `node_modules`, `dist` (если сборка на CI/Vercel), прочего мусора; `.gitignore` и история веток это гарантируют.
- Фронтенд без комментариев/блоков «режим разработки» и лишних dev-подсказок в прод-коде.
- Релевантный код просмотрен с фокусом senior-reviewer (архитектура, чистота, риски).
- Решение по `package.json` / lock / `requirements*.txt` задокументировано и реализовано так, чтобы **сборка Vercel оставалась рабочей** (Root Directory, install/build commands).

- Итог: осмысленные коммиты, PR `main` ← `dev` (или обоснованный обратный поток), чеклист «что отправить на проверку».

## Задачи (детали)

### REL-001 — `.gitignore`, политика артефактов и веток

- **Приоритет:** Critical  
- **Сложность:** Simple  
- **Файлы:** `.gitignore`, при необходимости `.cursor/workspace/.gitignore`  
- **Содержание:** гарантировать игнор: `ai_docs/` (для main — не в индексе; на dev может быть по политике команды), `.venv/`, `venv/`, `.pytest_cache/`, `node_modules/`, `frontend/dist/`, `**/__pycache__/`, `.env`, локальные state-файлы бэкенда (например `*.json` состояния watcher — по факту в репо).  
- **Критерии приёмки:** `git status` не показывает мусор после типичных команд; политика «что в main vs dev» зафиксирована в плане/чеклисте REL-009.

### REL-002 — Удаление лишних файлов из репозитория

- **Приоритет:** High  
- **Сложность:** Moderate  
- **Зависимости:** REL-001  
- **Содержание:** убрать из отслеживания и диска то, что не должно быть в git (кэши, сгенерированный `dist`, дубликаты путей Windows/Unix если есть); не трогать необходимые для сборки исходники.  
- **Критерии приёмки:** нет закоммиченных `node_modules`, `dist`, `.pytest_cache`; размер диффа осмысленный.

### REL-003 — Frontend: чистка dev-пояснений

- **Приоритет:** High  
- **Сложность:** Simple  
- **Файлы:** `frontend/src/**/*.tsx`, `frontend/src/**/*.ts`  
- **Содержание:** удалить комментарии и UI-тексты «для разработки», временные пояснения, отладочные подсказки; не ломать UX продукта.  
- **Критерии приёмки:** поиск по ключевым словам (dev, debug, TODO в UI) пуст или обоснован; линтер фронта без новых ошибок.

### REL-004 — Корень репо: зависимости и Vercel

- **Приоритет:** Critical  
- **Сложность:** Complex  
- **Зависимости:** REL-001  

**Рекомендация (для «чистого» main и рабочего Vercel):**

| Артефакт | Вариант A (предпочтительно для Vercel) | Вариант B |
|----------|----------------------------------------|-----------|
| `package.json` + lock | Оставить в **корне** репозитория на `main`: Vercel по умолчанию ожидает install в root; workspaces уже указывают на `frontend`. «Засорение» компенсируется минимальным корнем (только мета-пакет + README). | Перенести в подкаталог (например `tools/` или `repo-root/`) и в настройках Vercel задать **Root Directory** на этот каталог + скорректировать `workspaces` и пути в скриптах. |
| `requirements.txt` / `requirements-dev.txt` | Держать только в **`dev`** или в **`backend/`** на обеих ветках, но **не публиковать** в `main`, если Python на Vercel не используется. Если бэкенд не деплоится с фронта — на `main` можно исключить папку `backend` целиком (политика ветки). | Оставить в корне `dev`, в `main` удалить файлы — риск: проверяющий не увидит Python-часть на main. |

- **Критерии приёмки:** задокументирован выбранный вариант; `npm run build` из инструкции README проходит; в Vercel указаны корректные Root / Install / Build (если менялись).

### REL-005 — README для проверки

- **Приоритет:** High  
- **Зависимости:** REL-004, REL-006  
- **Содержание:** 1 экран: название, стек, как запустить фронт локально, как задеплоить на Vercel, где лежит исходник фронта; для `dev` — опционально ссылка на полную документацию в `ai_docs` (только в ветке dev).  
- **Критерии приёмки:** новый человек поднимает проект по шагам без угадываний.

### REL-006 — Структура веток `main` и `dev`

- **Приоритет:** Critical  
- **Зависимости:** REL-001  
- **Содержание:** `dev` содержит `.cursor/`, `ai_docs/`, dev-requirements, скрипты оркестрации; `main` — минимальное дерево: `frontend/`, корневые файлы сборки по решению REL-004, краткий README, лицензия при наличии, `.gitignore`. Слияние: типичный поток **PR `main` ← `dev`** с удалением из индекса на main лишнего (или **rebase dev на main** с последующим fast-forward main — выбрать одну стратегию без дублирования истории мусором).  
- **Критерии приёмки:** две ветки различаются предсказуемо; нет утечки секретов (`.env` вне git).

### REL-007 — Senior-reviewer по релевантному коду

- **Приоритет:** High  
- **Зависимости:** REL-003, REL-004, REL-006  
- **Область:** `frontend/src/**`, публичные точки входа, конфиги сборки, при наличии на ветке — `backend/*.py` (без секретов).  
- **Критерии приёмки:** список замечаний: Critical/High закрыты или заведены как issues; архитектура и чистота соответствуют KISS/SOLID для масштаба проекта.

### REL-008 — Тесты и сборка

- **Приоритет:** High  
- **Зависимости:** REL-002, REL-007  
- **Содержание:** `pytest` в корне (или `python -m pytest tests`); `npm ci` + `npm run build` в корне согласно финальной структуре.  
- **Критерии приёмки:** все тесты зелёные; сборка фронта успешна.

### REL-009 — Git, PR, инструкция проверяющему

- **Приоритет:** Medium  
- **Зависимости:** REL-008  
- **Содержание:** атомарные коммиты; открыть PR в GitHub; в описании PR — ссылка на оркестрацию, список изменений, как проверить (команды). Пользователю: «отправьте ссылку на PR в `main` + (опционально) ветку `dev` для `.cursor`».  
- **Критерии приёмки:** PR готов к review; чеклист в конце отчёта documenter.

## Риски

- Перенос `package.json` из корня без смены Root Directory на Vercel сломает деплой.

- Дубликаты путей (`backend\` и `backend/`) на Windows — унифицировать через git и `.gitignore`.
