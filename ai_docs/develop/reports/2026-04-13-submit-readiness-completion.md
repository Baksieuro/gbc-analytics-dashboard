## Итог senior-reviewer

- **RLS (Row Level Security):** зафиксированы **риски**, связанные с политиками доступа к данным на стороне БД/Supabase: важно явно проверять политики для ролей и сценариев дашборда, чтобы не было утечек между клиентами/пользователями при запросах из API и фронта.
- **Пагинация:** отмечены **риски** при больших объёмах (лимиты, курсоры/offset, согласованность с UI и бэкендом): при расширении нагрузки имеет смысл убедиться, что списки и агрегаты не режут данные молча и не создают лишнюю нагрузку без индексов.

*Детали замечаний и закрытие Critical/High — по issues/ревью в репозитории.*

## Сборка и проверка

| Команда | Назначение |
|---------|------------|
| `npm run build` | Сборка фронтенда из корня репозитория (согласно финальной структуре и `package.json`). |
| `pytest` | Python-тесты (из корня или по инструкции в `README`; типично после установки зависимостей из `backend/`). |

**Зависимости Python:** файлы **`requirements.txt` / `requirements-dev.txt`** располагаются в каталоге **`backend/`** (установка venv и `pip install -r …` — оттуда). На **`main`** бэкенд может отсутствовать по политике ветки; полный код и requirements — на **`dev`**.

---

## ⚠️ Слияние веток: нельзя «простой merge» `dev` → `main`

Ветки **намеренно различаются** по составу файлов (`main` — деплой/Vercel, `dev` — полный репозиторий с `ai_docs`, `.cursor`, бэкендом). **Прямой merge `dev` в `main`** может:

- затянуть в `main` каталоги и файлы, которые должны оставаться только на `dev`;
- нарушить минимальное дерево для Vercel или дублировать артефакты.

Используйте согласованную стратегию из плана (например, **PR с явным удалением из индекса** лишнего на стороне `main`, **rebase** или выборочное перенесение коммитов), а не слепой fast-forward без проверки дерева.
